## Лабораторная работа №1: Выявление и документирование требований
**Цель:** Освоить методы выявления и первичного анализа требований к программному продукту.

### Задание:
1. **Выбор предметной области:** Выбрать одну из предложенных (бронирование билетов, учет финансов, подбор мастеров и т.д.) или согласовать свою.
2. **Описание системы:** Определить назначение системы и целевых пользователей.
3. **Источники информации:** Определить не менее 3 источников и ожидаемую информацию от них.
4. **Анализ:** Зафиксировать 3–5 наблюдений или фактов о предметной области и поведении пользователе.
5. **Сценарии использования:** Описать 3–5 типовых сценариев в свободной форме (акторы + последовательность действий).
6. **Выделение требований:**
    * 10–15 функциональных требований (FR).
    * 4–6 нефункциональных требований (NFR).

## Лабораторная работа №2: Формализация и анализ качества требований
**Цель:** Оценка требований по критериям качества, группировка и описание Use Cases.

### Задание:
1. **Анализ качества:** Оценить требования из ЛР №1 по критериям: атомарность, выполнимость, полнота, краткость, недвусмысленность, консистентность, тестируемость.
2. **Группировка:** Разделить требования по логическим блокам (Управление пользователями, Основной функционал, Интеграции, Отчетность и т.д.).
3. **Use Cases:** Перевести сценарии из ЛР №1 в формат Use Case со ссылками на соответствующие требования.
4. **NFR и Архитектура:** Для каждого NFR определить атрибут качества и описать необходимые архитектурные решения.

## Лабораторная работа №3: Приоритизация и трассировка требований
**Цель:** Расстановка приоритетов, построение матрицы трассировки (RTM) и описание тестовых сценариев.

### Задание:
1. **MOSCOW:** Приоритизировать все требования с обоснованием.
2. **RICE:** Оценить по 5 требований из категорий *Must Have* и *Should Have* по формуле $RICE = \frac{Reach \times Impact \times Confidence}{Effort}$.
3. **Матрица трассировки (RTM):** Подготовить таблицу связей: `ID требования` — `Оценка MOSCOW` — `Use Case`.
4. **Тестовые сценарии:** Создать сценарии в формате **Given / When / Then** для проверки требований.

## Лабораторная работа №4: Управление изменениями
**Цель:** Научиться оформлять запросы на изменения (CR) и анализировать их влияние на проект.

### Задание:
1. **Оформление Change Requests (CR):** Подготовить три обязательных сценария:
    * **CR-001:** Авторизация через ГосМесседжер.
    * **CR-002:** Удаление одной функции из списка.
    * **CR-003:** Ужесточение показателей для одного NFR.
2. **Анализ влияния:** Для каждого CR определить затронутые требования, Use Cases, тесты и трудозатраты.
3. **Процесс:** Описать регламент фиксации, оценки и уведомления заинтересованных сторон об изменениях.
4. **Обновление RTM:** Актуализировать матрицу трассировки с учетом изменений.

## Лабораторная работа №5: Создание документа SRS
**Цель:** Систематизация всех наработок в итоговый документ Software Requirements Specification по стандарту IEEE 830.

### Структура документа:
* **Введение:** Назначение, область применения, термины.
* **Общее описание:** Контекст, пользователи, ограничения.
* **Требования:** Актуальные списки FR и NFR (с учетом ЛР №4), сгруппированные по блокам.
* **Интерфейсы:** Описание UI (макеты), API и программных интерфейсов.
* **Сценарии и Трассировка:** Таблица Use Cases и актуальная матрица RTM.
* **История изменений:** Таблица всех Change Requests.
